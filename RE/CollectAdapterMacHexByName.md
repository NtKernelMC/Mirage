# CollectAdapterMacHexByName

## Контекст
- Бинарь: netc.dll (base 0x10000000, size 0x65d000)
- Функция: 0x100686B0 (sub_100686B0), размер 0x292
- AGENTS.md в корне проекта не найден, поэтому ориентируюсь на запрос пользователя.

## Метод
- Просмотр декомпиляции через MCP (IDA Pro).
- Сопоставление обращений к GetAdaptersInfo и полям IP_ADAPTER_INFO по смещениям.

## Наблюдения
- Вход: `a1` — адрес выходного строкового объекта (3 dword), `a2` — объект, у которого вызывается метод по vtable+0x44 для получения `const char*` (IP-строка). См. 0x100686EC.
- Если IP-строка равна NULL, функция обнуляет выходную строку и возвращает `a1`. См. 0x100686FB–0x10068710.
- Дважды вызывает GetAdaptersInfo: сначала получает требуемый размер (ошибка 111), затем выделяет буфер и вызывает снова. См. 0x10068721, 0x1006873A, 0x100687B3.
- Проходит список адаптеров и сравнивает входной IP со строкой IP адаптера (strcmp), используя первый IP из IpAddressList. См. 0x100687DA.
- При совпадении формирует MAC как 12-символьную hex-строку (6 байт Address в IP_ADAPTER_INFO) через форматирование "%02x" и конкатенацию. См. 0x10068883.
- Конкатенация идет через `sub_1001D220`, которая не копирует байты напрямую, а кодирует их XOR-схемой: `dst[i] = src[i] ^ i ^ ((196*(4-(i&3)))/5) ^ 0x31 ^ (1<<(i&7))`. Это объясняет, почему строка в `a1` выглядит как «мусор» при прямом чтении. См. 0x1001D220.
- Макет выходной строки — три указателя (begin/end/capacity). Длина = `end - begin`. См. 0x1001D220 (использует `this[1] - *this`).
- Записывает результат в `a1`, освобождает буфер и возвращает `a1`. См. 0x100688DF, 0x10068929.

## Открытые вопросы
- Точный тип `a2` и его метод vtable+0x44 (вероятно, объект сетевого контекста, отдающий IP-адрес).
- Не обрабатываются дополнительные IP-адреса из цепочки IpAddressList.Next.

## Изменения в IDB
- Переименован `sub_1001D220` -> `SString_AppendEncodedBytes`.
- Комментарии в 0x1001D220 и 0x1006889D о XOR-кодировании строк.
